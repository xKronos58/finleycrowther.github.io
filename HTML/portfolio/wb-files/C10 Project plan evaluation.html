INTRODUCTION

This document evaluates the project plan of Wattbroker, an energy commodity trading application, by assessing the original Gantt chart and identifying areas for improvement. The evaluation focuses on critical paths, task allocations, and time management. Based on these insights, a revised Gantt chart has been created to address inefficiencies in the original plan.

PROJECT PLAN EVALUATION

The effectiveness of the original project plan is evaluated by comparing it to the actual execution timeline. The primary purpose of this assessment is to identify the shortcomings of the initial Gantt chart and provide an optimized version that better reflects the real-world progression of Wattbroker. Key aspects like resource allocation, scheduling flexibility, and response to unforeseen challenges are also evaluated.

ORIGINAL GANTT CHART ANALYSIS

The annotated version of the original Gantt chart (included in this folder) highlights specific inefficiencies and errors. While the project plan captured several essential elements, it failed to account for the iterative nature of some development tasks.

CRITICAL PATH, MILESTONES, AND DEPENDENCIES

The critical path of Wattbroker’s development was identified correctly in the original Gantt chart, especially for key deliverables like Algorithm Development and Graphing Systems. However, dependencies between tasks like Graph Renderer and Data Collection were not flexible enough to account for continuous adjustments, particularly when additional iterations of Graph Refinement were required. Due to to this there are some gaps in the original critical path within the development section so it has been revised to go through a different path.

TASK LENGTH

The length of several tasks was overestimated, especially in the initial analysis and planning stages. For instance, tasks like UI Prototyping and Feedback Incorporation took significantly less time than initially predicted. On the other hand, Graph Renderer Development required much more time, spanning across multiple phases of the project as it was continually refined and enhanced.

The underestimation of time for Data Collection also impacted the project’s timeline, as additional resources had to be reallocated to complete this critical task.

TASK TIME

Similarly, the time allocated for certain tasks, such as Algorithm Optimization and User Feedback Sessions, did not reflect actual working hours. For example, the First User Feedback was delayed due to the need for an earlier implementation of core features like Data Visualization. The real work often occurred closer to deadlines than scheduled, revealing a need for more realistic time management and more regular task monitoring.

CONCLUSION

Overall, the original project plan failed to fully account for the iterative nature of software development, particularly in areas that required continuous refinement, like Graph Renderer and Data Collection. The lack of flexibility and accurate time estimations led to inefficiencies. The modified Gantt chart, included in this folder, incorporates these lessons, reallocating time to critical tasks and ensuring that feedback cycles are integrated more effectively.

MODIFIED GANTT CHART

The modified Gantt chart reflects the more fluid, iterative process that occurred during Wattbroker’s development. Tasks such as Graph Renderer Refinement have been extended to reflect their continuous nature, and Feedback and Testing are more distributed across the timeline to allow for earlier and more frequent iterations.

DEVELOPMENT MODEL ANALYSIS

Wattbroker adopted a hybrid development model, blending Agile and V-Model methodologies. The Agile component was particularly effective during the development phase, allowing for flexibility in responding to evolving user needs, especially regarding algorithm improvements and graphing features. However, the V-Model’s structured approach was beneficial in ensuring a solid foundation during the initial Planning and Design phases.

The integration of Agile later in the project could have been introduced earlier to better handle dynamic requirements, such as those related to Data Collection. Furthermore, incorporating risk analysis—possibly through elements of the Spiral Model—during the Development phase could have mitigated unforeseen delays in graphing system refinements.

DOCUMENTATION OF MODIFICATIONS MADE TO THE PROJECT PLAN

Several key modifications were made during the Wattbroker development cycle, particularly in response to the challenges faced with Graph Renderer and Data Collection. Both tasks took more time than anticipated, requiring ongoing adjustments throughout the development process.

For instance, after receiving feedback on early versions of the application, a decision was made to reallocate resources to extend the development timeline for Algorithm Tuning and Graphing Systems. Milestone adjustments, such as shifting Feedback after Submission to later stages, ensured that user expectations were met while still accommodating the continuous improvement of core features.
