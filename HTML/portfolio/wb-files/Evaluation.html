1. Evaluation Strategies for Efficiency and Effectiveness

When evaluating Wattbroker, it is crucial to focus on both the application side and the algorithm this allows the evaluation to be broken down into several different categories such as Algorithm efficiency, user interface and graphing / data representation. The chosen strategies to asses the most vital areas of Wattbroker are based around providing a high-performance experience for energy commodity traders.

  1. Algorithm Efficiency evaluation:
    • Performance Benchmarks Due to the high volatility of energy the performance of the algorithm is crucial, the algorithm needs to perform quickly under intense circumstances. These tests will be used to evaluate how quickly the algorithm can understand and predict data for energy spot price and usage statistics.
    • Stress Testing On top of performance testing stress testing also needs to be run to ensure that the algorithm can also perform with a high accuracy under may high loads and large datasets at peak times.
    • Accuracy and profitability metrics (Efficiency) Feeding historical data to the algorithm to observe the accuracy of the predictions can allow for more fine tuning of the algorithm. This can be used to measure its effectiveness on modern market graphs. Another method could be to allow the algorithm to predict with current market data however not give it the ability to actually decrement the balance however it would be able to decide how much to buy and sell.
  1. User interface (UI) Evaluation:
    • User testing and heat-maps will be used to get insight on how intuitive and easy to understand the user interface is, the user heat-maps can be conducted directly in FIGMA via the Attention Insight plugin. This will help identify bottlenecks and areas of confusion.
    • User satisfaction surveys will be able to capture feedback from users and further help understand what areas cause confusion and help understand the usability of the UI from those less technical users who have not before used the UI.
    • **MORE*
  1. Graphing and data representation Evaluation:
    • Rendering performance is crucial to the graphing system in allowing it to update without causing any noticeable lag to lower end systems

2. Evaluation against requirements

Algorithm

The energy trading algorithm is one of the most vital parts of the software solution (Wattbroker) and below are the criteria based off the functional and non-functional requirements rating its effectiveness and efficiency.

Language

As outlined in FR03 the algorithm should be written in a low level GPU based programming language such as CUDA or Intel One. A high-level result of this in the final would have the algorithm function in such a lower level language and harness the full potential of that language.

Performance

As outlined in FR03 the algorithm should also be able to be highly efficient when performing calculations, fully harnessing the potential of a GPU to give the most accurate results. A high-level result in performance would allow the algorithm to execute large datasets (1,000-10,000) calculations in >10ms. This, however, is separate from API response time as that will be stored on the server and only changed in use once attained.

Renewable Capabilities

Using weather data the algorithm should be able to fully predict the efficiency of major renewable farms such as solar or wind within the state that has been defined for it to operate on this would be some form of total output calculation. A high-level result to this would allow the algorithm to accurately calculate energy output of any given large renewable farm and semi-accurately predict the production of the farms.

Fossil Capabilities

As fossil is harder to predict and more closely relate to the usage of energy this would mean that the algorithm may not directly be predicting the fossil or it would be based off the renewable amounts, therefore, a high-level in this would show that the algorithm can relate some other dataset to assume the value of the fossil fuel.

Usage Capabilities

Based off weather and time the algorithm should be able to estimate the energy usage of a state, in doing so the algorithm can compare these values to the assumed production values and calculate the current and future spot price * referenced in the next paragraph. A high-level result in this category would show that the algorithm can reliably understand and predict the current usage of energy in a predefined state currently and in the future.

Spot Price predictions

As the most crucial part of the algorithm is to be able to predict the spot price of the energy market and use those pointers to be able to “buy low and sell high”. To do this the algorithm needs to be able to incorporate all of the previous criteria to allow it to accurately estimate or predict the spot price. A high-level result of this would mean that the algorithm can reliably predict the spot price using the other types of prediction to create a final result

User Interface (UI)

The user interface for Wattbroker was one of the most vial components to having a functional app and setting it apart from its competitors.

Intuitive

As outlined by NFR02 the UI of Wattbroker must be intuitive, as Wattbroker is a platform that allows for both entry level and experts in the energy commodity trading sector it is vital that the design is intuitive for all in the sector. To assess a high-level result of this in the final solution user testing and design heat-mapping should be incorporated. From user testing a high-level result would create a “ease-of-use” score of < 3.5/5.

Clean and thought-out

Another similar requirement for the UI is that it is clean and well-made / thought out. This is crucial to the user experience of Wattbroker so that users are not left guessing what elements are and why design choices were made. A high-level of this would include a result from user testing showing that the design was easy to navigate and that their overall satisfaction with the design is high (4+/5).

Design scalability

As users will be using the application on several different resolutions and aspect ratios the design should also be scalable, this is outlined in NFR04. To get a high-level result in this the design should automatically scale to whatever dimensions the user provides.

Key interfaces

In FR07-11 several key interfaces are described (Login, Dashboard, Algorithm, Market, Data) these menus are necessary for Wattbroker to be able to perform its core functions without any of these interfaces Wattbroker would only be partially functional. For the final product to produce a high-level outcome with this it would have to have all of the mentioned interfaces to a fully functional standard.

Data

As the Wattbroker algorithm is built off datasets and would not function without them they highly crucial. The below described criteria are what will be used to measure the accuracy and effectiveness of the data collection and handling methods.

Identification

Without relevant data the algorithm would not produce a relevant result. Listed in FR01 the application should be able to identify and attain relevant datasets. A high-level implementation of this in the final application would incorporate relevant API calls from sources such as AEMO (Market) and BOM (weather).

Data Storage

With such large sets of data, a storage server should be used to allow the algorithm to communicate with a storage server containing several months’ worth of data to run historical training simulations. A high-level implementation of these protocols would incorporate a server (same or separate to algorithm server) that is capable of storing large sets of data that can communicate with the client and algorithm.

Custom data management

As the algorithms are so complicated and have such a high degree of customisation they cannot be stored within the code as this would drastically reduce overall performance. This means that a custom file type should be developed that can be easily read and modified by the code & user but still efficient in storage. A high-level result of this would include a template for the file type and an implementation of that file type reading.

Data refresh rate

When the algorithm creates computations, it requires up to date data, thus the data stored on the server should be updated as frequently as possible. To do this before each computation a check must be run to see if the data can be updated and if so, it should. A high-level outcome of this would involve a data refresh check built into the application that the user can customise that directly corresponds with the algorithm.

User Handling

As each person using Wattbroker must have a user account it is important that it has been implemented. The below sections will aid in evaluating the effectiveness of the user handling system.

Volume

The client AFC industries currently have quite a small employee size thus the user volume handling does not need to be able to support millions of users but should be able to support the current employee base + 50% for the immediate future as shown by FR05. This allows all employees to have an account with the provided software and allows the company to grow by at least 50% without having to make modifications to Wattbroker. A high-level outcome for this would be demonstrated by the application being able to handle more than base + 50% users without any performance or data issues.

Security

As each user will have a password, they must be adequately hashed encase of the event of a data breach leading to the SQL database being exposed. If the passwords are not adequately hashed and protected this would go against the Privacy act of 2001. A high-level outcome for this would show that Wattbroker has adequately protected and hashed passwords to protect against data breaches.

Account creation

Each user must be able to create an account and password on their own to allow them to user Wattbroker. A high-level result of this would show that Wattbroker is able to let users sign up and sign in with the newly created account.

Configurability and Compatibility

As Wattbroker will be used by a group of people with a diverse set of skills and needs on a large range of devices, Wattbroker will need to be compatible with the different devices and should be configurable to each specialists needs to most efficiently complete their work.

Configurability

As stated above there will be a diverse set of highly skilled professionals alongside users who are still learning, due to this the platform should be configurable to each set of skills to allow for all users to perform with the most ease and efficiency whilst using Wattbroker. A high-level outcome from this would show that the user can customise the platform via the settings and customise the UI to how they want it.

Operating system compatibility

As not all specialists will have the same device running the same operating system, this must be accounted for. The client-side application should be able to run on popular Linux distributions, macOS and windows to allow for the widest range of systems to be used to run the client-side application. For the server-side code however, this will not be used on such a wide range of systems due to some of the other requirements and the hardware requirements. A high-level outcome of this would demonstrate that Wattbroker can operate on all major operating systems (Popular Linux distributions, MacOS, & Windows) with a range of different hardware, the server-side code is somewhat exempt from this requirement as it requires specific NVidia hardware and is built for UBUNTU server and WINDOWS server.

Hardware compatibility

As Wattbroker should be compatible with several different operating systems it should also be compatible (client-side) with most hardware between low and high end. The server application requires specific hardware although that is due to a language requirement. To do this the client-side application should be optimised so that it can run on low end systems and should be able to perform the same. A high-level result of this would be the client-side being able to run on an intel Pentium silver or equivalent as very few machines have slower hardware than that so it would be unreasonable to require anything faster.

5. Impact of the development model

The hybrid Agile-V model used for Wattbroker’s development has had a significant impact on both the development process and the final product. By combining the flexibility of Agile (Scrum) with the structured approach of the V-Model, I was able to adapt to evolving requirements while maintaining rigorous testing and validation protocols at every stage.

The Agile (Scrum) components allowed for iterative feedback loops, ensuring that features were continuously tested and refined based on client input. This flexibility helped prioritise high-value features and adjust algorithms in real-time, providing greater adaptability for forecasting energy prices and responding to market fluctuations. Additionally, the Agile sprints ensured timely delivery of essential modules, such as the algorithm integration and advanced graphing capabilities.

On the other hand, the V-Model enforced a disciplined, step-by-step approach to development. Each development phase, from requirements gathering to system design, was paired with a corresponding testing phase. This structure ensured that no stage was skipped or rushed, leading to a thoroughly tested and validated product. However, with the intense time constraints of this model there it was not always followed as if it was the project would not have been completed by the deadline*(reword). The combination of these methodologies created a balance between rapid iteration and careful, controlled progress, leading to the successful delivery of a robust and user-friendly trading platform.

4. Critical and creative thinking during the development cycle

There were many instances of critical and creative thinking being used throughout the development cycle of Wattbroker. During the project there were several complex problems and challenges, especially around the Graphing system and the algorithm.

For the algorithm, critical and creative thinking was applied to ensure that the prediction model was not only accurate but efficient and flexible enough to handle various types of input data. This includes weather data, market data and sever others to fully understand the current real world situation. To continuously asses the accuracy the algorithm would be set to predict current values based off historical (outdated) data. This rigorous analysis ensured that the algorithm was adequately tested under a variety of market conditions.

Creative thinking was also crucial during the development of the graphing system as it needs to handle several different datasets and render at different sizes. During the development many different approaches were utilised but had to be changed due to performance considerations. Initially the graph was made by using a map of coordinates and then mapping a Java-FX point to each coordinate based off its value. That was really in-performant due to the amount of nodes being rendered on the larger graphs and lead to gaps when the accuracy of the points was not high enough. To get around this a java-FX cubic curve group was used, three points of values then create a curve to those points. Rather than mapping a co-ordinate list it would scale based off x and y values and the points would correspond to pixel x and y.

The interaction between critical analysis and creative problem-solving throughout the cycle led to innovative solutions, such as real-time graph updates and algorithm tuning, which are crucial to Wattbroker’s functionality0.
